11. FLUJO DE TRABAJO (desde el móvil, sin programar)
================================================================

El usuario trabaja desde el móvil. NO escribe código a mano.
El flujo es siempre el mismo:

1) Usuario detecta un bug o pide una funcionalidad nueva
   conversando con Claude (este Claude, en la app).

2) Claude diagnostica, escribe los archivos arreglados, los
   empaqueta (uno suelto o un ZIP) y devuelve UN ÚNICO MENSAJE
   con dos cosas:
     - El archivo o ZIP adjunto.
     - Un PROMPT listo para copiar-pegar al agente.

3) El usuario copia ese prompt y lo pega en el agente que
   actualiza el repo desde el móvil (Replit, Netlify, o el que
   esté usando en ese momento).

4) El agente coge el archivo adjunto, lo mete donde toca en el
   repo, actualiza el CHANGELOG, hace commit con el mensaje
   exacto que le dimos, y hace push.

5) GitHub Pages redespliega automáticamente en pocos minutos.
   El usuario abre la URL del juego y verifica.

QUÉ SIGNIFICA ESTO PARA CLAUDE
------------------------------
- El prompt debe ser AUTÓNOMO. El agente que lo recibe no tiene
  contexto previo. Todo lo que necesita debe estar en el prompt.
- Las rutas de los archivos deben ser EXACTAS y completas
  (ej. `css/13_barra_skip.css`, no "el archivo del skip").
- No asumas que el agente "ya sabe" la estructura del repo.
  Si hay duda, explícala.
- Si un cambio toca varios archivos, lístalos TODOS.

13. AVISOS Y ERRORES TÍPICOS DEL AGENTE
================================================================

Los agentes que actualizan el repo (Replit y similares) tienen
manías documentadas. Tenlas presentes:

1) MIENTEN A VECES.
   Dicen "está hecho" cuando solo lo han cambiado en su entorno
   local y no han hecho push al repo de GitHub. Si el usuario
   reporta que algo no se ve aplicado, verificar el historial
   de commits en GitHub antes de asumir que el fix está mal.

2) HACEN COSAS DE MÁS POR SU CUENTA.
   A veces fusionan archivos, renombran, o aplican fixes
   adyacentes que no estaban en el prompt. Suele ir bien, pero
   conviene revisar el historial de commits después de cada
   entrega.

3) GITHUB CACHEA.
   Si haces web_fetch al repo justo después de un push, puedes
   ver la versión vieja. Esperar unos minutos o pedir al usuario
   una captura del listado real de archivos.

4) NO ASUMIR QUE EL FIX HA LLEGADO AL JUEGO.
   Que esté en el repo no significa que esté en producción
   inmediata. GitHub Pages tarda 1-5 minutos en redesplegar.
   El móvil del usuario puede tener caché. Recargar sin caché
   o en modo incógnito.

5) LÍMITES DE USO.
   El agente puede quedarse sin cuota. Si el usuario dice "he
   llegado al límite", parar y esperar / cambiar de herramienta.

15. PLANTILLA DEL PROMPT PARA EL AGENTE
================================================================

El prompt debe seguir EXACTAMENTE esta estructura, en este orden,
en español, en markdown (los **negritas** importan, el agente
las usa para parsear):

---PLANTILLA INICIO---

**Tarea:** [una frase clara que resume qué hay que hacer]

**Contexto:** [2-3 líneas sobre qué hacía mal y qué hace bien
ahora. Solo si aporta. Si es un fix obvio, omitir esta sección.]

**Pasos:**

1. [Reemplazar / crear / mover archivos. Indicar RUTAS EXACTAS
   desde la raíz del repo, ej. `css/13_barra_skip.css`.]

2. [Si toca varios archivos, listarlos TODOS, uno por línea.]

3. Actualiza `CHANGELOG.md`. Añade esta entrada **arriba del
   todo** (justo debajo del título `# CHANGELOG` si lo hay):

   ```
   ## [0.XX] - YYYY-MM-DD
   ### Fixed / Changed / Added / Removed
   - [descripción en una línea, en español, sin tecnicismos
     innecesarios, entendible por una persona no técnica]
   ```

4. Commit con este mensaje exacto:

   ```
   tipo(area): descripción corta en minúsculas

   Cuerpo opcional de 2-4 líneas explicando el qué y el porqué.

   v0.XX
   ```

5. Push a la rama actual.

**No toques nada más.**

---PLANTILLA FIN---

NOTAS SOBRE LA PLANTILLA
------------------------
- Fecha en formato YYYY-MM-DD (ej. 2026-05-26).
- La línea `v0.XX` al final del cuerpo del commit es OBLIGATORIA.
  Es como el agente etiqueta la versión.
- El bloque de commit va dentro de triple backtick para que el
  usuario lo pueda copiar pulsando el botón de copiar.
- El changelog también va en triple backtick por la misma razón.
- "No toques nada más" al final es importante: evita que el
  agente se ponga creativo y modifique archivos no incluidos.

TIPOS DE COMMIT PERMITIDOS
--------------------------
- fix(area)      — corrige un bug
- feat(area)     — nueva funcionalidad
- refactor(area) — reorganización sin cambio funcional
- style(area)    — cambios visuales / CSS sin lógica
- docs(area)     — solo documentación
- chore(area)    — mantenimiento, dependencias, configuración

Ejemplos de `area`: `intro`, `mara`, `audio`, `ui`, `assets`,
`barra`, `mercado`, `persistencia`, `launcher`.

TIPOS DE ENTRADA EN CHANGELOG
-----------------------------
- ### Fixed   — bugs corregidos
- ### Changed — comportamiento que cambia
- ### Added   — cosas nuevas
- ### Removed — cosas eliminadas
- ### Refactored — reorganización interna sin cambio visible

- Cuerpo opcional de 2-4 líneas explicando QUÉ y POR QUÉ
  (nunca CÓMO — eso ya está en el diff).
- Línea en blanco.
- Última línea: `v0.XX` con la versión correspondiente.

ARCHIVOS — REGLAS DE ORO
------------------------
- Rutas SIEMPRE desde la raíz del repo. Sin "./" delante.
- No mover archivos sin avisar explícitamente en los pasos.
- No renombrar archivos a no ser que la tarea sea exactamente

- Respetar la numeración de los prefijos (01_, 02_, ..., 30_).
